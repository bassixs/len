## Дальнейший план (roadmap)

Кратко — проект уже приведён в “чистое” состояние: единая модель товара, выделенный сервис корзины, общий UI-слой, ESLint/Prettier, форматированный код. Ниже — приоритетные шаги развития.

### 1. Корзина и чекаут (продолжение)
- **Сделать UX чекаута ещё понятнее**
  - Явно подсвечивать бесплатную доставку (например, текстом под суммой доставок, когда `delivery === 0`).
  - Показать “осталось до бесплатной доставки” при сумме меньше порога (5000 ₽).
- **Согласовать поля cart-item с бэком**
  - Если в будущем появится серверная часть, договориться о схеме (`id`, `sku`, `selectedSize`, `selectedColor`, `quantity`, `price`) и привести локальную модель к ней.

### 2. Каталог: фильтры и сортировка
- **Реальные фильтры на основе `data/products/*.json`**
  - Добавить фильтрацию по:
    - цене (диапазон),
    - категориям/подкатегориям,
    - размеру/цвету (если есть в данных).
  - Сохранять состояние фильтров в URL (`?priceFrom=...&size=...`), чтобы страницы можно было шарить.
- **Сортировка**
  - По цене (возрастание/убывание), новизне, популярности (при наличии поля).
  - Отдельный UI-элемент сортировки с явной привязкой к данным.

### 3. Product-page и карточки товаров
- **Связать статику с данными**
  - Убрать хардкоды текста/цен в `product.html` и “похожие товары” подбирать по категории/тегам.
  - Добавить в `product-model` дополнительные поля (например, “материал”, “плотность”, “stock”), если они появятся в JSON.
- **Улучшить выбор опций**
  - Чётко подсвечивать недоступные размеры/цвета (при появлении информации о наличии).
  - При смене размера/цвета обновлять цену/картинку, если в данных предусмотрены варианты.

### 4. UL/UX-улучшения
- **Единые состояния кнопок**
  - Использовать `btn--submitting`, `btn--secondary` и другие утилитарные классы вместо инлайн-стилей.
- **Информативные тосты и ошибки**
  - Отдельные типы уведомлений: успех, ошибка, предупреждение (цвета/иконки).
  - Понятные сообщения при ошибках загрузки данных (каталог, карточка товара).

### 5. Тесты и стабильность
- **Первые юнит-тесты для чистых функций**
  - `normalizeProduct`, `formatPrice`, `getDeliveryCost`, `getCartTotals`, `pluralize`.
  - Запуск через лёгкий тестовый раннер (например, Vitest), без тяжёлой конфигурации.
- **Smoke-тесты UI (минимум)**
  - Проверить, что инициализация модулей (`initCart`, `initCatalog`, `initProductPage` и т.д.) не падает на пустых/альтернативных страницах.

### 6. Потенциальная интеграция с бэком
- **API-слой (при появлении сервера)**
  - Ввести модуль `api.js` с обёрткой над `fetch`: базовый URL, обработка ошибок, типы запросов.
  - Перевести загрузку данных товаров с `public/data/...` на реальный API, не меняя остальной код (через адаптер).
